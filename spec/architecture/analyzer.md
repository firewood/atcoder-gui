# OpenSpec: Analyzer 詳細設計 (atcoder-gui)

## 役割 (Role)
`analyzer` コンポーネントは、AtCoderの問題ページから得られる非構造的な情報を、プログラムで扱いやすい構造化データへと変換する役割を担います。単なる文字列抽出にとどまらず、入力形式の構文解析や、サンプル入力に基づいた変数の型推論を行うことで、高精度なコード生成の基盤を提供します。

## 主要コンポーネント

### 1. HTML Parser (`src/analyzer/html-parser.ts`)
Cheerioを使用して問題ページのHTMLを解析し、以下の情報を抽出します。
- **入力形式 (Input Format)**: `<pre>` タグ内に記述された入力の構成。
- **サンプル入力/出力**: `Sample Input` / `Sample Output` セクションから得られる具体的な値のペア。
- **特殊フラグ**: 
    - `multipleCases`: `T` 個のテストケースが含まれる形式かどうかを判定。
    - `queryType`: クエリ形式の問題（例：クエリ数 $Q$ とそれに続く $Q$ 行の命令）かどうかを判定。

### 2. Lexer & Parser (`src/analyzer/lexer.ts`, `src/analyzer/parser.ts`)
抽出された「入力形式」の文字列を解析し、初期のAST（抽象構文木）を構築します。
- **Lexer**: 数値、変数名、ドット（...）、演算子、括弧などのトークンに分割。
- **Parser**: トークン列から `FormatNode`, `ItemNode`, `BinOpNode` などのノードを生成し、階層構造を構築。

### 3. Analyzer (`src/analyzer/analyzer.ts`)
生のASTを正規化し、意味的な解釈を加えます。
- **ループ検出 (Loop Detection)**: 
    - 「$A_1, A_2, \dots, A_N$」のような記述から、`A` がインデックス `i` を持ち、`1` から `N` まで繰り返されるループ構造であることを特定。
    - `dotsIndex` を中心に左右のパターンを比較し、一致する場合に `LoopNode` を生成。
- **変数名の生成**: ループ変数として `i`, `j`, `k` などの適切な名前を自動割り当て。

### 4. Typing Engine (`src/analyzer/typing.ts`)
ASTと実際のサンプル入力を照合し、各変数のデータ型を推論します。
- **型推論ロジック**:
    - 全てのサンプル入力において数値のみであれば `ValueInt` または `Float`。
    - 1文字であれば `Char`、それ以外は `String`。
    - インデックスとして使われている場合は `IndexInt`。
- **型の統合 (Unification)**: 複数のサンプルで異なる型（例：ある例では整数、別の例では小数）が現れた場合、より広範な型（この場合は `Float`）に統合。
- **ループの畳み込み (Loop Collapsing)**: インデックス付きの変数が1次元配列として扱われている場合、解析を容易にするために内部的に構造を簡略化。
    - **2Dグリッドの1D化**: スペースを含まない2Dグリッド形式（例：$S_{i,j}$ が連なっている記述）において、サンプル入力にスペースが含まれない場合、内側のループを畳み込み、1次元の文字列（`String`）配列として扱う。これにより、`cin >> S[i]` のような簡潔なコード生成を可能にする。

### 5. Matcher (`src/analyzer/match.ts`)
ASTの構造に従って、実際の入力文字列から変数の値を抽出します。型推論はこのマッチャーの実行結果を利用して行われます。

## 処理フロー

1. **抽出**: HTMLから `inputFormat` と `samples` を取り出す。
2. **構築**: `inputFormat` をトークン化し、初期ASTを作成。
3. **洗練**: `Analyzer` がループ構造を検出し、ASTを論理的な構造に変換。
4. **推論**: 
    - 各 `samples.input` に対して AST を使って値を抽出。
    - 抽出された全ての値から、各変数の最適な型を決定。
5. **出力**: 構造化された `FormatNode` と変数メタデータのリストを `generator` に渡す。

## 設計上の特徴

- **ロバスト性**: 正規表現によるパターンマッチングに頼らず、言語処理系に近いアプローチ（Lexer/Parser）を採用しているため、複雑な入力形式にも対応可能。
- **実証的推論**: 問題文の説明だけでなく、実際の入力データ（サンプル）を確認することで、変数の型（特に `int` か `double` か）を正確に判断。
- **拡張性**: `analyzer` の出力は言語非依存のメタデータであるため、新しいプログラミング言語のジェネレータを容易に追加可能。
